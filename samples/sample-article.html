Authenticating with the Payments API

How to obtain, use, and refresh an access token for the Payments API.

  • Audience: Backend developers integrating a third-party payments API.
  • Goal: Make a first authenticated request within ten minutes.
  • My role: Sole author — research, drafting, engineering review, publication.
  • Tools: Markdown, OpenAPI, Git, Docusaurus.

Every request to the Payments API must include a valid access token. This page explains how to exchange your client credentials for a token, attach it to requests, and handle expiry. Replace this whole page with one of your own samples — the structure below is a reusable skeleton for reference documentation.

Before you begin

You need the following:

  • A merchant account with API access enabled.
  • A client ID and client secret from the Developers → Credentials page.
  • An HTTPS-capable client. Plain HTTP requests are rejected.

Note: Sandbox and production use separate credentials. A sandbox secret will return 401 invalid_client against the production host.

Step 1 — Request an access token

Send your credentials to the token endpoint using the client_credentials grant. 

curl -X POST https://api.example.com/oauth/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials" \
  -d "client_id=$CLIENT_ID" \
  -d "client_secret=$CLIENT_SECRET" \
  -d "scope=payments.read payments.write"

A successful request returns 200 OK:

{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "payments.read payments.write"
}

Step 2 — Call the API

Include the token in the Authorization header of every request. 

curl https://api.example.com/v1/payments \
  -H "Authorization: Bearer $ACCESS_TOKEN"

Step 3 — Refresh before expiry

Tokens are valid for one hour. Cache the token and request a new one when fewer than five minutes remain, rather than requesting a token per API call — the token endpoint is rate-limited to 60 requests per minute per client.

Scopes

Scope Grants Required by
payments.read Read payments and refunds GET /v1/payments
payments.write Create and capture payments POST /v1/payments
refunds.write Issue refunds POST /v1/refunds

Error reference

Status Code Fix
400 unsupported_grant_type Set grant_type to client_credentials.
401 invalid_client Check the credentials match the environment you are calling.
403 insufficient_scope Request the scope listed for that endpoint above.
429 rate_limited Cache tokens and retry after the Retry-After interval.
